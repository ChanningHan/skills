# 技术方案评审 Skill

## 概述

本 Skill 为 AI Agent 提供技术方案评审能力，指导 Agent 从多个专业角色视角评审技术方案文档，并生成综合评审报告。

### 核心能力

1. **多角色视角评审**：从前端、后端、产品、交互等不同专业视角评审技术方案
2. **结构化评审流程**：遵循标准化的评审流程，确保评审全面性
3. **综合评审报告**：汇总各角色评审意见，生成结构化的评审报告

### 适用场景

- 技术方案设计评审
- 架构设计评审
- 产品功能方案评审
- 系统升级改造方案评审

## 评审流程

Agent 应按照以下标准流程进行技术方案评审：

### 1. 读取技术方案文档

- 使用 `read_file` 工具读取技术方案文档
- 理解方案的整体结构和内容
- 识别方案的关键章节（背景、目标、架构设计、技术选型、实施计划等）

### 2. 分析方案结构

- 提取方案的核心要素：
  - 背景与目标
  - 技术架构设计
  - 技术选型
  - 实施计划
  - 风险评估
- 识别各角色的评审要点

### 3. 角色评审

依次从以下角色视角进行评审，每个角色独立评审：

- 前端专家 1
- 前端专家 2
- 后端专家
- 产品经理
- 交互设计师

**重要**：Agent 需要为每个角色单独进行评审，从该角色的专业视角出发，提供具体的评审意见。

### 4. 汇总评审意见

- 整理各角色的评审意见
- 识别共性问题和分歧点
- 提取关键优势和主要担忧
- 整理可行的改进建议

### 5. 生成评审报告

- 按照标准格式生成评审报告
- 保存评审报告到指定路径

## 角色定义与评审指南

### 前端专家（Frontend Expert）

**角色定位**：资深前端技术专家，熟悉前端架构设计、性能优化、用户体验实现。

**关注领域**：
1. 前端技术架构设计
2. 代码质量和可维护性
3. 性能优化
4. 用户体验实现
5. 前端技术选型

**评审维度**：

#### 1. 架构合理性
- 组件划分是否合理？
- 状态管理方案是否恰当？
- 代码组织结构是否清晰？
- 模块间耦合度是否合理？

#### 2. 代码可维护性
- 是否遵循编码规范？
- 代码复杂度是否可控？
- 是否有充分的注释和文档？
- 是否考虑了代码复用？

#### 3. 性能考虑
- 是否有性能优化方案？
- 渲染性能是否考虑充分？
- 是否有懒加载、代码分割等优化？
- 是否考虑了首屏加载速度？

#### 4. 用户体验实现
- 交互实现是否符合设计规范？
- 是否考虑了边界情况和错误处理？
- 是否有加载状态和反馈？
- 响应式设计是否考虑充分？

#### 5. 技术选型合理性
- 技术栈选择是否合适？
- 第三方库选择是否恰当？
- 是否考虑了技术债务？
- 是否有技术升级计划？

**评审输出格式**：
```markdown
### 前端专家 [编号] 评审意见

#### 评审维度评价

##### 1. 架构合理性 [评分：1-5]
- [具体评价]
- [优势]
- [问题]
- [建议]

##### 2. 代码可维护性 [评分：1-5]
- [具体评价]
- [优势]
- [问题]
- [建议]

[... 其他维度 ...]

#### 整体评价

**优势**：
- [优势 1]
- [优势 2]

**担忧**：
- [担忧 1]
- [担忧 2]

**建议**：
- [建议 1]
- [建议 2]

**综合评分**：[1-5 分]
```

### 后端专家（Backend Expert）

**角色定位**：资深后端技术专家，熟悉系统架构设计、接口设计、数据模型设计。

**关注领域**：
1. 接口设计
2. 数据模型设计
3. 系统架构
4. 安全性
5. 后端技术选型

**评审维度**：

#### 1. 接口设计合理性
- 接口设计是否符合 RESTful 规范？
- 接口粒度是否合适？
- 是否考虑了接口版本管理？
- 是否有完善的接口文档？
- 错误处理是否统一？

#### 2. 数据模型设计
- 数据模型设计是否合理？
- 是否满足业务需求？
- 是否考虑了数据扩展性？
- 是否有数据一致性保障？

#### 3. 系统架构
- 系统架构是否合理？
- 是否考虑了高可用和容错？
- 是否有扩展性考虑？
- 是否考虑了性能瓶颈？

#### 4. 安全性考虑
- 是否有权限控制方案？
- 是否考虑了数据安全？
- 是否有防范常见安全漏洞？
- 是否有日志和审计？

#### 5. 技术选型合理性
- 后端技术栈选择是否合适？
- 数据库选型是否恰当？
- 中间件选择是否合理？
- 是否考虑了运维和监控？

### 产品经理（Product Manager）

**角色定位**：资深产品经理，熟悉业务需求、产品设计、用户体验。

**关注领域**：
1. 业务价值
2. 用户需求满足度
3. 功能完整性
4. 优先级合理性
5. 实施可行性

**评审维度**：

#### 1. 业务价值
- 方案是否解决了核心业务问题？
- 是否有明确的业务目标？
- 预期收益是否清晰？
- 是否考虑了 ROI？

#### 2. 需求满足度
- 是否满足用户核心需求？
- 是否覆盖了关键使用场景？
- 是否有需求遗漏？
- 是否有过度设计？

#### 3. 功能完整性
- 功能设计是否完整？
- 是否考虑了边界情况？
- 是否有用户反馈机制？
- 是否有数据分析支持？

#### 4. 优先级合理性
- 实施计划的优先级是否合理？
- 是否区分了 MVP 和后续迭代？
- 是否考虑了快速验证？
- 资源分配是否合理？

#### 5. 实施可行性
- 方案是否可行？
- 是否考虑了技术限制？
- 是否有清晰的交付标准？
- 是否有风险应对方案？

## Agent 使用指南

### 完整评审流程示例

```markdown
1. 读取技术方案文档
   - 使用 read_file 读取技术方案文档
   - 理解方案的整体结构

2. 分析方案结构
   - 识别关键章节和评审要点
   - 为每个角色准备评审上下文

3. 执行角色评审（按顺序进行）
   
   a. 前端专家 1 评审
      - 从前端专家视角审视方案
      - 关注前端架构、代码质量、性能、用户体验实现、技术选型
      - 按照评审维度逐一评价
      - 输出结构化评审意见
   
   b. 前端专家 2 评审
      - 从另一位前端专家视角审视方案
      - 可能关注不同的细节或有不同的观点
      - 按照评审维度逐一评价
      - 输出结构化评审意见
   
   c. 后端专家评审
      - 从后端专家视角审视方案
      - 关注接口设计、数据模型、系统架构、安全性、技术选型
      - 按照评审维度逐一评价
      - 输出结构化评审意见
   
   d. 产品经理评审
      - 从产品视角审视方案
      - 关注业务价值、需求满足度、功能完整性、优先级、可行性
      - 按照评审维度逐一评价
      - 输出结构化评审意见
   
   e. 交互设计师评审
      - 从交互设计视角审视方案
      - 关注交互流程、用户体验、界面设计、易用性、用户旅程
      - 按照评审维度逐一评价
      - 输出结构化评审意见

4. 汇总评审意见
   - 整理各角色的评审意见
   - 识别共性问题和分歧点
   - 提取关键优势和主要担忧
   - 分类整理改进建议

5. 生成评审报告
   - 按照标准格式组织评审报告
   - 生成行动项清单
   - 输出评审结论
   - 保存评审报告到指定路径
```

### 注意事项

1. **角色独立性**：每个角色的评审应该独立进行，从该角色的专业视角出发
2. **评审深度**：评审应该具体、详细，避免泛泛而谈
3. **建设性意见**：不仅要指出问题，还要提供改进建议
4. **客观公正**：基于事实和专业知识进行评审，避免主观臆断
5. **结构化输出**：严格按照定义的格式输出评审意见
