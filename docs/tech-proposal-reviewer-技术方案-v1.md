# tech-proposal-reviewer Skill 技术方案

## 一、背景

### 1.1 业务需求

在技术方案评审过程中，需要从多个专业角色视角（前端、后端、产品、交互设计）进行全面评审，确保方案的技术可行性、业务价值和用户体验。传统的人工评审方式存在以下问题：

- **评审效率低**：需要多个专家参与，协调成本高
- **评审标准不统一**：不同评审者关注点不同，评审结果难以标准化
- **评审报告格式不统一**：缺乏标准化的评审报告格式，难以对比和分析

### 1.2 解决方案

通过 AI Agent Skill 的方式，标准化技术方案评审流程，指导 AI Agent 从多个专业角色视角进行评审，并生成结构化的评审报告。

## 二、设计思路

### 2.1 核心设计理念

1. **多角色视角评审**：模拟真实评审场景，从 5 个专业角色（前端专家 2 位、后端专家 1 位、产品经理 1 位、交互设计师 1 位）独立评审
2. **结构化评审流程**：定义标准化的评审流程，确保评审的全面性和一致性
3. **标准化输出格式**：统一的评审报告格式，便于对比和分析

### 2.2 架构设计

```
tech-proposal-reviewer/
├── SKILL.md                    # Skill 主文档（评审指南）
├── LICENSE.txt                  # 许可证
├── examples/                    # 示例文件
│   └── sample-review-report.md # 评审报告示例
└── references/                  # 参考资料
    └── review-template.md      # 评审报告模板
```

**设计要点**：
- **文档驱动**：通过 Markdown 文档定义评审流程和标准，无需代码实现
- **模板化**：提供评审报告模板和示例，确保输出格式统一
- **可扩展**：通过文档扩展即可支持新的评审角色或维度

## 三、实现细节

### 3.1 评审流程

Agent 按照以下流程执行评审：

1. **读取技术方案文档**：使用 `read_file` 工具读取待评审的技术方案
2. **分析方案结构**：提取核心要素（背景、架构、技术选型、实施计划等）
3. **角色评审**：依次从 5 个角色视角独立评审
4. **汇总评审意见**：整理各角色意见，识别共性问题和分歧点
5. **生成评审报告**：按照标准格式生成评审报告并保存

### 3.2 角色定义

每个角色都有明确的：
- **角色定位**：专业背景和职责范围
- **关注领域**：5 个评审维度
- **评审维度**：每个维度的具体评审要点
- **输出格式**：标准化的评审意见格式

**角色列表**：
- 前端专家 1：关注架构、代码质量、性能、用户体验、技术选型
- 前端专家 2：从不同视角补充前端评审
- 后端专家：关注接口设计、数据模型、系统架构、安全性、技术选型
- 产品经理：关注业务价值、需求满足度、功能完整性、优先级、可行性
- 交互设计师：关注交互流程、用户体验、界面设计、易用性、用户旅程

### 3.3 评审报告格式

评审报告包含以下章节：

1. **评审概述**：方案基本信息、评审范围、评审结论
2. **各角色评审意见**：5 个角色的完整评审意见
3. **评审汇总**：关键优势、主要担忧、改进建议
4. **行动项**：优先级排序的改进建议清单
5. **评审结论**：整体评价、通过建议、前置条件、后续建议

### 3.4 技术实现

**实现方式**：
- **纯文档实现**：通过 Markdown 文档定义评审指南，无需代码
- **Agent 执行**：AI Agent 读取 Skill 文档，按照指南执行评审
- **工具调用**：Agent 使用 `read_file`、`write` 等工具完成评审任务

**优势**：
- 实现简单，维护成本低
- 易于扩展和修改评审标准
- 不依赖特定技术栈

## 四、可能的风险点

### 4.1 评审质量风险

**风险**：AI Agent 可能无法完全理解技术方案的复杂细节，评审深度不够

**应对措施**：
- 提供详细的评审维度和要点指导
- 通过示例报告展示评审深度要求
- 鼓励 Agent 多次阅读方案文档，深入理解

### 4.2 评审一致性风险

**风险**：不同 Agent 执行评审时，评审标准理解不一致

**应对措施**：
- 提供标准化的评审模板和示例
- 明确定义每个角色的评审维度和评分标准
- 通过示例报告展示评审格式要求

### 4.3 扩展性风险

**风险**：新增评审角色或维度时，需要修改多个文档

**应对措施**：
- 采用模块化的文档结构
- 角色定义独立，便于新增
- 评审报告模板支持灵活扩展

## 五、后续优化方向

1. **评审深度增强**：增加更多评审维度和检查清单
2. **评审自动化**：提供脚本工具辅助 Agent 执行评审
3. **评审结果分析**：提供评审结果的统计和分析功能
4. **评审知识库**：积累常见问题和最佳实践
